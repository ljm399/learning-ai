# 基础ai对话项目的优化 -- 流式传输

### 之前后端直接返回json，优化为流式传输

#### 优势（为什么要做流式）

- 解释
  - **更快的“首字延迟”体验**
    - JSON：必须等模型整段回答生成完，接口一次性返回
    - 流式：模型边生成边推送，前端可以逐字/逐段渲染
  - **更适合长文本输出**
    - 输出越长，“等待整段返回”的体感越差，流式能显著改善体验
  - **更容易做“打断/停止生成”**（成熟产品常见）
    - 前端断开 SSE 连接即可停止接收（服务端也能感知并中止）

#### 什么时候用 JSON，什么时候用流式

- 使用 JSON（一次性返回）更合适
  - 输出很短（如分类/打标/判断）
  - 需要一次性拿到结构化结果（JSON Schema 等），中途不希望渲染半成品
  - 接口链路里还要做较多后处理（比如要等多个工具调用完成后再返回）
- 使用流式更合适
  - 需要“像打字一样”的对话体验（聊天/写作/代码生成）
  - 输出较长、或不确定会输出多久
  - 想实现“边生成边展示、随时停止”



## SSE技术

#### 解释

##### 1）SSE 是什么：基于 HTTP 的“单向推送”

- 解释
  - SSE（Server-Sent Events）是一种**服务端 -> 浏览器**的推送协议
  - 底层仍然是 **HTTP 长连接**：客户端发起一次请求，服务端不立刻结束响应，而是持续往响应里写入数据
  - 非常适合大模型：模型边生成边推送，前端边接收边渲染（“打字机效果”）

##### 2）SSE 和 JSON 一次性返回 / WebSocket 的区别

- 解释
  - JSON 一次性返回
    - 一次请求一次响应
    - 简单，但必须等到“最终答案”才能展示
  - SSE
    - 一次请求，多次推送（服务端多次 `res.write`）
    - 更轻量，足够支撑“流式输出”场景；但它是**单向**（服务端 -> 客户端）
  - WebSocket
    - 全双工（客户端与服务端都能随时发消息）
    - 更适合强实时双向交互；只做“流式输出”时往往 SSE 更省事

##### 3）SSE 消息格式：`data:` + 内容 + 空行（`\n\n`）

- 解释
  - SSE 事件是一行行文本构成的
  - 最常用字段是 `data:`，后面一般放字符串；工程里通常放 `JSON.stringify(...)`
  - 一条事件必须以空行结束（也就是 `\n\n`），客户端才会触发一次 [onmessage](cci:1://file:///c:/Users/MJL/Desktop/ai%E5%AD%A6%E4%B9%A0/ppt%E5%92%8C%E6%BA%90%E7%A0%81/1-9%20%E6%B5%81%E5%BC%8F%E4%BC%A0%E8%BE%93/code/aiclient/src/api/index.js:15:8-17:9)
- 代码（对应你后端的写法：`ppt和源码/1-9 流式传输/code/server/index.js`）

  ```js
  // 必须以 data: 开头，并以 \n\n 结束
  res.write(`data:${JSON.stringify(resObj)} \n\n`)
  ```

#### 前端接收：原生 `EventSource` vs `fetch-event-source`

- 解释
  - 原生 `EventSource(url)`
    - 优点：浏览器内置
    - 限制：只能 `GET`，自定义 header / 传 body 不方便
  - `@microsoft/fetch-event-source`
    - 本质仍然是 SSE（服务端响应仍是 `text/event-stream`）
    - 但客户端发起请求走 `fetch`：支持 `POST`、自定义 header、JSON body（与你的项目实现一致）

## 实战

### 大概过程

### 1）接口侧把调用大模型的地方加上 `stream: true`

- 解释
  - `openai.chat.completions.create({ stream: true })` 会返回一个可迭代的流
  - 你需要 `for await...of` 逐个拿到 chunk
- 进一步解释（为什么这里能写 `for await (let chunk of llmres)`）
  - `await` **不只**能写在“发网络请求”前面
  - `await` 的本质：等待一个 **Promise** 兑现
  - `for await...of` 的本质：用来遍历 **异步可迭代对象**（AsyncIterable）
    - 同步遍历用：`for (const x of iterable)`（iterable 提供 `Symbol.iterator`）
    - 异步遍历用：`for await (const x of asyncIterable)`（asyncIterable 提供 `Symbol.asyncIterator`）
  - 在本项目里：
    - `await openai.chat.completions.create({ stream: true })` 这一步返回的 `llmres` 不是“最终回答字符串”，而是一个**可以不断产出 chunk 的异步流对象**
    - 所以你后面才能用 `for await...of` 一段段消费它（每次循环拿到一个 `chunk`）
  - 常见误解澄清
    - 误解：`await` 只能写在网络请求前
    - 实际：任何返回 Promise 的事情都可以 `await`，例如：读文件、定时器、数据库、以及这里的“从流里等下一段数据”
- 代码（`ppt和源码/1-9 流式传输/code/server/index.js`）

  ```js

      model: "qwen-plus",
      messages: [
          {
              role: "system",
              content: systemString
          },

      stream: true
  })s

  }
  ```

#### `for await...of` 的等价理解（帮助你形成直觉）

- 解释
  - `for await...of` 你可以把它理解成：每次循环都在“等下一段数据到来”
  - 更底层一点，它相当于不断调用异步迭代器的 `next()` 并 `await` 结果
- 代码（伪等价写法，帮助理解；项目里并不需要这样写）

  ```js
  const it = llmres[Symbol.asyncIterator]();
  while (true) {
      const { value, done } = await it.next();
      if (done) break;
      const chunk = value;
      // ... 处理 chunk
  }
  ```

### 2）修改响应头为 SSE

- 解释
  - `Content-Type: text/event-stream` 告诉客户端这是 SSE
  - `keep-alive` 保持连接不断开
- 代码（`ppt和源码/1-9 流式传输/code/server/index.js`）

  ```js
  res.writeHead(200, {
      'Content-Type': 'text/event-stream; charset=utf-8',
      'Cache-Control': 'no-cache',
      'Connection': 'keep-alive',
  });
  ```

### 3）用 `res.write` 按固定格式一段段写入（而不是一次性 `res.json`）

- 解释
  - 每个 chunk 的核心增量内容在：`chunk.choices[0].delta.content`
  - 服务端做了一个“累加对象”`resObj`：
    - `id` 固定为当前流的 `chunk.id`
    - `content` 每次拼接 `delta.content`
    - 每次都把“最新的完整对象”推给前端
- 代码（`ppt和源码/1-9 流式传输/code/server/index.js`）

  ```js
  let resObj = {
      role: "assistant",
      id: "",
      content: ""
  }
  
  for await (let chunk of llmres) {
      const delta = chunk.choices[0].delta
      resObj.id = chunk.id
      resObj.content += delta.content
      res.write(`data:${JSON.stringify(resObj)} \n\n`)
  }
  ```

### 4）流式结束：写入 done 信号 + `res.end()`

- 解释
  - for-await 循环结束，说明模型生成完了
  - 这时：
    - 把完整回答写入会话记录（持久化）
    - 给前端发一个 `{ done: true }` 作为结束标记
    - `res.end()` 关闭 SSE
- 代码（`ppt和源码/1-9 流式传输/code/server/index.js`）

  ```js
  // 每次回答，存到 singleConvertList，保存上下文
  singleConvertList.push(resObj);
  writeConversation(conversationObj)
  res.write(`data:${JSON.stringify({ done: true })} \n\n`)
  res.end()
  ```

### 1）每次收到 SSE 消息，把 `event.data` 解析成对象

- 解释
  - 服务端写的是 `data:${JSON.stringify(resObj)}`
  - 前端 `event.data` 就是这段 JSON 字符串
- 代码（`ppt和源码/1-9 流式传输/code/aiclient/src/views/Home.vue`）

  ```js

  })
  ```

### 2）用 `id` 做“同一条 assistant 消息”的定位：有则替换，无则新增

- 解释
  - 流式每个 chunk 都会推一个“同 id 的对象”
  - 前端要做的是：
    - 第一次收到该 id：push 一条 assistant 消息到 `convertList`
    - 之后每次收到该 id：找到数组里同 id 的那条，替换成最新内容（实现“逐字更新”）
- 代码（`ppt和源码/1-9 流式传输/code/aiclient/src/views/Home.vue`）

  ```js
  const assistantObj = JSON.parse(event.data)
  
  const _convertList = [...convertList.value];
  const convertIndex = _convertList.findIndex(item => item.id === assistantObj.id);
  if (convertIndex !== -1) {
      _convertList[convertIndex] = assistantObj
  } else {
      isThinking.value = false;
      _convertList.push(assistantObj)
  }
  convertList.value = _convertList;
  ```

### 3）接口处理逻辑

- 解释
  - 服务端每次 chunk 取出 `delta.content`，不断拼接到 `resObj.content`
  - 客户端根据 `id` 找到同一条消息并替换，让 UI 看起来是“一个气泡在增长”
  - 循环结束后，服务端把完整回答写入历史（会话持久化），前端刷新后可通过 `conversation/get` 拿到完整记录





# function tool

## 目标：让模型“选工具 + 传参数”，大模型就是脑子但是没手，这里当作其手

- **解释**
  - Function Tool（也叫 Tool Calling / Function Calling）的核心是：
    - 你先把“可用工具清单（函数名/用途/参数）”发给大模型
    - 模型在需要时返回 `tool_calls`（告诉你要调用哪个函数、参数是什么）
    - 服务端根据 `tool_calls` 去执行你自己的业务代码（订票/查天气/打车…）
    - （完整形态）再把工具执行结果发回模型，让模型生成最终自然语言回复

## 步骤 1）定义工具清单 `tools`（告诉模型：有哪些函数可用）

- **解释**
  - 工具清单是一个数组，每个元素描述一个工具
  - 本项目使用 `type: "function"`，并在 `function` 字段里声明：
    - `name`：函数名（模型返回的 `tool_calls.function.name` 就会用它）
    - `description`：模型什么时候应该调用
    - `parameters`：参数结构描述（用于让模型产出结构化 arguments）
- **代码**（`ppt和源码/2-1 function tool是什么/code/server/tools.js`）

```js
const toolList = [
    {
        type: "function",
        function: {
            name: "help_dp",
            description: "当用户需要订票的时候调用此工具",
            parameters: {
                city: {
                    type: "string",
                    description: "用户要去的城市"
                }
            }
        }
    }
]
```

## 步骤 2）把 `tools` 随着对话一起发给大模型

- **解释**
  - 你在 `openai.chat.completions.create(...)` 里传入 `tools: toolList`
  - 模型就“知道你提供了哪些工具”，从而在合适时机返回 `tool_calls`
- **代码**（`ppt和源码/2-1 function tool是什么/code/server/index.js` 的 `/llm`）

```js

        {
            role: "system",
            content: systemString
        },
```

## 步骤 3）流式模式下：从 `delta` 里“拼接出完整 tool_calls”

- **解释**
  - 流式返回不是一次性给完整 JSON，而是一段段 chunk
  - `tool_calls` 在流式里也会“分片返回”，所以你必须像拼接 `content` 一样拼接：
    - `function.name`
    - `function.arguments`
  - 本项目用 `index` 来定位某一个 tool call（一个回答可能包含多个工具调用）
- **代码**（`ppt和源码/2-1 function tool是什么/code/server/index.js`）

```js
for await (let chunk of llmres) {
    const delta = chunk.choices[0].delta
    resObj.id = chunk.id
    resObj.content += delta.content || ''
    if (delta.tool_calls && delta.tool_calls.length > 0) {
        //拼接tool_calls部分
        if (resObj.tool_calls) {
            //已经是第一个以后的chunk，走拼接
            delta.tool_calls.forEach((chunkTool) => {
                const toolIndex = chunkTool.index;
                //根据index找到resObj，要拼接进去的对象
                const targetTool = resObj.tool_calls[toolIndex]
                if (chunkTool.function?.name) {
                    targetTool.function.name += chunkTool.function?.name
                }
                if (chunkTool.function?.arguments) {
                    targetTool.function.arguments += chunkTool.function?.arguments
                }
            })

        } else {
            //你还是第一个chunk,直接走赋值
            resObj.tool_calls = delta.tool_calls;
        }
    } else {
        res.write(`data:${JSON.stringify(resObj)} \n\n`)
    }
}
```

## 步骤 4）本项目当前行为：检测到 `tool_calls` 时先不往前端推（先把结构拼好）

- **解释**
  - 你在 `else` 分支里才 `res.write(...)`
  - 也就是：
    - **普通文本增量**：直接推给前端（SSE 实时渲染）
    - **工具调用增量**：先在服务端把 `tool_calls` 拼接完整（暂时不推）
- **代码**（同上：`res.write(...)` 只出现在 `else` 分支）

```js
if (delta.tool_calls && delta.tool_calls.length > 0) {
    // ... 拼接 tool_calls
} else {
    res.write(`data:${JSON.stringify(resObj)} \n\n`)
}
```

## 步骤 6）准备工具执行入口：`toolHandleMap`（把“函数名”映射到真实业务逻辑）

- **解释**
  - 模型返回的 `tool_calls.function.name` 本质就是字符串
  - 服务端一般会做一个映射表（白名单）：只允许调用你显式声明过的函数
  - 本项目已经准备了 `toolHandleMap`，用于把工具名映射到实现函数
  - 注意：这里有一个明显的拼写不一致：
    - 工具名是 `help_dp`
    - 映射表里写成了 `hepl_dp`
    - 如果你后续真去用 `toolHandleMap[toolName]` 执行，会找不到
- **代码**（`ppt和源码/2-1 function tool是什么/code/server/tools.js`）

```js
const toolHandleMap = {
    hepl_dp() {
        //订票的具体逻辑
    },
    help_dc() {
        //打车的逻辑
    }
}
```

## 步骤 7）完整的 function tool 调用闭环应长什么样

- **解释**
  - 你现在已经做到了：
    - `tools` 传入模型
    - 流式把 `tool_calls` 拼接出来
  - 完整闭环应该是：
    - 流式结束后检查 `resObj.tool_calls`
    - 遍历每个 tool call
    - 取出 `function.name` 和 `function.arguments`
    - `JSON.parse(arguments)` 得到工具入参
    - 用 `toolHandleMap[name]` 执行真实代码拿到 `result`
    - 构造一条 `role: "tool"` 的消息，把工具执行结果发回给模型
    - 再次调用模型，让它基于工具结果输出“给前端展示的最终答复”

## 2-2 项目：服务端执行工具 + 把工具结果再发回模型

#### 步骤 7.1）流式结束后：检查是否有 `tool_calls`

- **解释**
  - 在 `for await...of` 结束后，`resObj` 已经拼出了完整 `tool_calls`
  - 先把本轮 assistant 消息写入会话历史
  - 如果存在 `tool_calls`，就进入“执行工具”分支；否则正常返回 `{ done: true }`
- **代码**（`ppt和源码/2-2 执行Function tool/code/server/utils.js`）

```js
singleConvertList.push(resObj);
writeConversation(conversationObj)

if (resObj.tool_calls && resObj.tool_calls.length > 0) {
    const tool_calls = resObj.tool_calls
    // ... 执行工具
} else {
    res.write(`data:${JSON.stringify({ done: true })} \n\n`)
    res.end()
}
```

#### 步骤 7.2）遍历每个 tool call：按 `name` 找到方法并执行

- **解释**
  - `singleTool.function.name` 是模型选择的工具名
  - `singleTool.function.arguments` 是**JSON 字符串**，需要 `JSON.parse`
  - 执行时用映射表 `toolHandleMap[name]`（白名单思想）
- **代码**（`ppt和源码/2-2 执行Function tool/code/server/utils.js`）

```js
for (let toolIndex = 0; toolIndex < tool_calls.length; toolIndex++) {
    const singleTool = tool_calls[toolIndex];
    const name = singleTool.function.name;
    const arguments = JSON.parse(singleTool.function.arguments);
    const result = await toolHandleMap[name](arguments)
    // ... 把 result 回传给模型
}
```
